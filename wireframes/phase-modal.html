01 What's crowding the current row

The row carries five separate visual elements before it says anything about the chapter itself. Two of them contradict each other, and one repeats on eight of eleven rows.

Current — as shipped640px

Transistors & Logic Gates Build Python Read

123 45
  • 1

    Two "done" languages disagree. Every row shows a filled blue square — that's content status (the lesson is written). The header right above says 0 / 11 completed — that's your progress. A wall of blue squares over a 0% bar reads as a bug.

  • 2

    A label dressed as a button. LEARN/BUILD is a category, but it's boxed exactly like READ, which is the only real action. Same visual weight, opposite grammar — so the eye can't find the one thing to click.

  • 3

    A column that repeats itself. "Python" appears on 8 of 11 rows and "—" on the rest. It costs a full column and separates the title from its action.

  • 4

    A long dead gap. Title on the far left, controls on the far right, nothing between. On a 640px modal the eye travels the full width to bind a name to its buttons.

  • 5

    No sense of weight. Nothing distinguishes a 782-word chapter from one nearly twice as long. That's the gap all three variants close.

Give the numbers their own line so they inform without competing. The whole row becomes the click target, so no button is needed at all.

  • Row is one link — READ button removed entirely
  • LEARN/BUILD demoted to a text label, not a box
  • Language dropped from the row (it belongs on the lesson page)
  • Content-status square removed; the checkbox is now the only "done" mark
  • Phase totals added at the top
Trade-off Roughly doubles row height — all 11 chapters no longer fit without scrolling, which is exactly when size comparison is hardest.

BSpec sheet

Turn the list into an index: fixed columns, a bar scaled to the longest chapter in the phase. Relative size is read pre-attentively — before you read a single number.

  • Bars scale from zero, so length is honest, not exaggerated
  • Fixed columns kill the dead gap — nothing floats
  • L/B tag sits with the title instead of in its own column
  • Words and minutes both shown; digits use tabular figures so they align
  • All 11 chapters visible at once — the comparison actually works
Trade-off The densest of the three, and the bar spends horizontal room that long chapter titles may want. Titles need truncation at narrow widths.

The tightest option — a printed table of contents. Leader dots carry the eye from title to number, and a five-segment meter stands in for the bar.

  • Shortest rows of the three — the most chapters per screen
  • Leader dots solve the dead gap without adding a column
  • Segmented meter reads as "size bucket" rather than false precision
  • Familiar as a book's contents page — fits the manual identity
Trade-off Five segments is coarser than a continuous bar — chapters 05 and 06 (935 vs 957 words) look identical. Fine for triage, useless for ranking.

05 Recommendation

Ship B — Spec sheet

It's the only one that answers the actual question at a glance. Because the bar is scaled to the phase's longest chapter, "which is big and which is small" is legible before you read a digit — and all 11 rows still fit, so the comparison happens in one view instead of across a scroll.

B already absorbs every cleanup from A: no READ button, no language column, no content-status square fighting the progress checkbox. If B tests too dense with real users, A is the safe fallback — it keeps the cleanups and trades the bar for breathing room.

06 Words, not lines

You mentioned lines or words. Words are the better signal: source lines count formatting, not reading effort — blank lines, code fences and table pipes all inflate a chapter that reads quickly. Words map to a minutes estimate, which is what someone deciding "do I start this now?" actually wants.

Note the spread is real but modest — 782 to 1,465 words, a 1.9× range. That's an argument for a bar scaled to the phase maximum (as in B): scaled to an absolute ceiling, every chapter would look the same length.

To build this: data.js carries no counts today. build.js already reads each docs/en.md, so it's a small addition there — emit words per lesson and a phase total, derive minutes at words / 200. Nothing else in the pipeline changes.

On the type: these frames use your token system, but published pages can't load font CDNs — so VT323 and Source Serif fall back to system faces here. In the real modal the headings will be noticeably more characterful.
