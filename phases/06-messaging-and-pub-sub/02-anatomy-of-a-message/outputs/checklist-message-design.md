# Checklist — Designing a Message Contract

Run this once per **message type**, before the first producer ships. A message contract outlives
the team that wrote it: consumers you have never met will parse these bytes years from now, and
you cannot ask them to redeploy in lockstep. Getting the envelope right is cheap now and
impossible later.

Paste the completed **contract record** at the bottom into your design doc.

## Step 1 — Separate the envelope from the payload

- [ ] Write one sentence naming the **business fact**, in the past tense: *"an order was placed"*,
      *"a payment was authorized"*. If you cannot, you may be designing a command, not an event —
      decide which, and name it accordingly (`order.place` vs `order.placed`).
- [ ] Everything in that sentence is **payload**. Everything about *the message carrying it* is
      **envelope**.
- [ ] Apply the routing test to every field you are unsure about: **will anything other than the
      final consumer need to read this?** If yes it belongs in the envelope (as a header); if no
      it belongs in the body.
- [ ] Confirm no infrastructure component needs to parse the body to do its job. Routing,
      deduplication, ordering, retry, dead-lettering, audit and tracing must all be answerable
      from envelope fields alone.
- [ ] For each field you **promote** from body to envelope, record why. Promotion is not free: it
      becomes part of the infrastructure contract, it duplicates data that can drift from the
      body, and it enlarges every message.

## Step 2 — Envelope fields present, and each one justified

Tick each field or write down the failure you are accepting by omitting it.

| Field | Present? | The failure it prevents |
|---|---|---|
| `message_id` | ☐ | A redelivery is indistinguishable from a new event → the retry double-charges |
| `correlation_id` | ☐ | You cannot gather the messages belonging to one business transaction |
| `causation_id` | ☐ | You have a bag of related messages but no idea what caused what |
| `type` | ☐ | The consumer guesses meaning from the body's shape |
| `schema_version` (or `dataschema`) | ☐ | One schema change breaks every consumer at once |
| `source` | ☐ | Nobody knows who to page when the payload is wrong |
| `occurred_at` | ☐ | You cannot tell a replay from a live event |
| `published_at` | ☐ | Producer-side lag is invisible |
| `recorded_at` (broker-stamped) | ☐ | You have no clock you did not have to trust |
| `content_type` | ☐ | The consumer sniffs the format |
| `content_encoding` | ☐ | A compressed body is handed to a JSON parser |
| `traceparent` | ☐ | The async hop is a black hole in your traces |
| `partition_key` | ☐ | Related events are processed out of order |
| Integrity digest | ☐ | Truncation and corruption are applied rather than detected |

## Step 3 — Id strategy

- [ ] `message_id` is generated by the **producer, before the first send attempt**, and is
      **reused unchanged on every retry**. An id generated per network attempt deduplicates
      nothing.
- [ ] The id is globally unique without coordination — a UUID (RFC 4122, updated by RFC 9562),
      not a per-service counter.
- [ ] Consider **UUIDv7** rather than v4 for `message_id`. Both are unique; v7 prefixes a
      millisecond timestamp so ids sort by creation time, which turns your dedup table's index
      from random B-tree inserts into append-mostly ones and makes expiry a range delete.
- [ ] `correlation_id` is created **once**, at the edge of the business transaction, and copied
      unchanged onto every descendant message however deep.
- [ ] `causation_id` is set to the **incoming message's `message_id`** — the immediate parent,
      not the root. It changes at every hop.
- [ ] A root message (no parent) has `causation_id` absent or null, and consumers handle that.
- [ ] You can run the **orphan query**: messages whose `causation_id` names no message you hold.
      That single join finds lost publishes.

## Step 4 — Time semantics

- [ ] No field is named just `timestamp`. Every time field says *whose* clock and *which* moment.
- [ ] `occurred_at` is **event time** — when the fact happened in the world. Business logic and
      analytics use this one and no other.
- [ ] `published_at` is **processing time** on the producer side. Expect it to lag `occurred_at`
      by hours for mobile clients, outbox drains and retry storms.
- [ ] `recorded_at` is stamped by the **broker**, not the producer. It is the only timestamp that
      does not depend on a clock you do not control.
- [ ] Time units and epoch are documented and identical across all three (e.g. microseconds since
      Unix epoch, UTC). Mixed seconds/milliseconds is a recurring production bug.
- [ ] If anything aggregates by time window, you have decided **which** timestamp buckets it, and
      how long you wait for late data before closing a window.
- [ ] Your broker's native timestamp field is mapped explicitly. (Kafka: `CreateTime` is the
      producer's, `LogAppendTime` is the broker's — pick per topic and write it down.)

## Step 5 — Format choice, decided on two axes

Do not pick from a menu. Place yourself on both axes and let the answer fall out.

| | Self-describing | Schema-required |
|---|---|---|
| **Text** | JSON, XML — greppable, zero coordination, verbose | JSON + JSON Schema — validates but does not shrink |
| **Binary** | MessagePack, CBOR (RFC 8949) — most of the size win, no registry | Protobuf, Avro, Thrift — smallest, needs a registry |

- [ ] **Who debugs this at 3 a.m., and with what tool?** If the answer is "a human with `curl`",
      text wins regardless of size.
- [ ] **What is the volume?** Compute `bytes/message x messages/second x 86,400`. If that number
      is not uncomfortable, format size is not your problem — pick for ergonomics.
- [ ] **Can you run a schema registry?** Schema-required formats are not adoptable without one,
      plus a compatibility policy (see lesson 12). If you cannot commit to operating it, choose
      self-describing.
- [ ] **Does the payload contain binary data?** JSON has no byte type; base64 costs a flat +33%.
- [ ] **JSON number trap checked**: any integer that can exceed 2^53−1 (9,007,199,254,740,991) —
      64-bit database ids, Snowflake ids, nanosecond timestamps — is carried as a **string**.
      IEEE 754 doubles silently corrupt them otherwise.
- [ ] You **measured with your own messages** rather than trusting a benchmark. The ratio is
      dominated by how much of your bytes are field names and identifiers.

## Step 6 — Size budget and the claim-check threshold

- [ ] Know your broker's hard limit: **SQS 256 KiB · Kafka ~1 MiB default (`message.max.bytes`) ·
      Google Pub/Sub 10 MiB · RabbitMQ frame limits in the hundreds of MiB**.
- [ ] Set your **own** threshold well below it — a small multiple of your p99 message size, not
      the broker's ceiling.
- [ ] Know your p99 and p100 message size. Unbounded arrays in a body (line items, tags, events)
      are where p100 escapes: cap them, or the pattern below is mandatory.
- [ ] Above the threshold, apply the **claim-check pattern**. The ticket carries: `uri`,
      **content digest**, `bytes`, `content_type`, `content_encoding`, `expires_at`.
- [ ] The consumer **verifies the digest** after fetching. Without it the pointer is an
      unvalidated download.
- [ ] The blob is written **before** the message is published. Never the reverse.
- [ ] Storage lifecycle ≥ **max queue retention + max dead-letter age + replay window**, with
      margin. If you replay from a log, the blob must still exist.
- [ ] You have decided who deletes the blob after successful consumption, and what happens if
      nobody does. Both dangling pointers and leaked blobs are on someone's runbook.

## Step 7 — Compression decision

- [ ] Compression is applied to **batches**, not to individual messages. Compressing one small
      message routinely *inflates* it — the codec's own header outweighs the redundancy available.
- [ ] There is a size floor below which you skip compression entirely (a few KB is typical).
- [ ] If your format is already schema-ful binary, expect a much smaller gain — the repeated field
      names that make text compress well are already gone.
- [ ] You checked the batching knob (`linger.ms` or equivalent). Waiting a few milliseconds to
      fill a batch often halves bandwidth; confirm the added latency is acceptable for this
      message type.
- [ ] `content_encoding` declares the codec, so a consumer never guesses.
- [ ] A **decompressed size limit** is enforced. A 1 KB body that expands to 10 GB is a
      decompression bomb.

## Step 8 — Validation, in this order

Every check below runs on the **envelope**, before any body parsing.

- [ ] `content_type` checked against an **allowlist**, never a denylist.
- [ ] `schema_version` checked against a supported range; a version from the future is rejected,
      not best-effort parsed.
- [ ] Ids parse as real UUIDs, of the expected variant.
- [ ] Body size checked **before** any buffer is allocated for it.
- [ ] Integrity digest verified before parsing.
- [ ] Time sanity checked (e.g. `published_at >= occurred_at`); a violation means clock skew or a
      forged event.
- [ ] Only then: deserialize into a **plain data structure**, validate against the declared
      schema, and only then construct domain objects. **Never let the wire choose the type** —
      that is what makes `pickle`, Java serialization and `BinaryFormatter` remote code execution.
- [ ] Depth, array-length and string-length limits are enforced during parsing.
- [ ] A validation failure routes **straight to the dead-letter queue with the reason attached**.
      It is not retried: a malformed message will fail identically forever (lesson 08).

## Step 9 — Immutability and correction

- [ ] Nothing in your design updates a published message. It is a fact about the past.
- [ ] There is a defined **correction path**: a new event (`order.amount.corrected`) or a
      compensating pair, each with its own `message_id` and a `causation_id` pointing at what it
      corrects.
- [ ] Consumers are safe to process the same `message_id` twice (lesson 06 makes this concrete).
- [ ] Nothing assumes a message will be redelivered with different content than the first time.

## Step 10 — Trace propagation

- [ ] The producer writes the current span context into the message as `traceparent`
      (W3C Trace Context), plus `tracestate` if you use vendor-specific state.
- [ ] The consumer **reads it and continues the trace** — as a child span, or as a span **link**
      when one consumer span covers a batch of differently-parented messages.
- [ ] `traceparent` is in the **envelope/headers**, not the body, so middleware can read it
      without parsing.
- [ ] You have verified end to end that a single trace crosses the async hop. Untested
      propagation is almost always broken propagation.
- [ ] `correlation_id` is carried **independently** of the trace id. Traces get sampled away;
      the business correlation must survive that.

## The contract record

Write this into the design doc so the decision survives the people who made it:

```text
MESSAGE      com.shop.order.placed          schema_version 1
owner        team-orders                    source urn:svc:orders
payload      order_id, customer_id, currency, amount_minor, items[]  (items capped at 200)
envelope     message_id(uuidv7) correlation_id causation_id type schema_version source
             occurred_at published_at recorded_at content_type content_encoding
             traceparent partition_key crc32
partition by customer_id            -- ordering scope, see lesson 07
format       <json | msgpack | protobuf>   because <debuggability | volume | registry available>
             measured: ___ B/msg avg, ___ B p99  ->  ___ GB/day at ___ msg/s
size budget  p99 ___ KB, threshold ___ KB, claim-check above it -> s3://___  lifecycle ___ days
compression  batch-level <codec>, floor ___ KB, linger ___ ms
validation   allowlist content_type; versions [___]; digest verified; DLQ on failure, no retry
correction   via com.shop.order.amount.corrected, causation_id -> the corrected message
```
