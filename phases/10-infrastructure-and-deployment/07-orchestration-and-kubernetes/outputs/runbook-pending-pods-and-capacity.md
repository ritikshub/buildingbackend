# Pending pods & capacity — triage runbook

Use this when the cluster is not running what you declared: replicas stuck `Pending`, a
node failure that has not recovered, or a drain that will not finish.

**The rule that saves the most time: the scheduler has already told you the answer.**
`Pending` is not a failure state — it means the scheduler ran, evaluated every node, found
none feasible, and put the work back on the queue. Nothing crashed, nothing is in a
back-off loop, and it will be scheduled the instant a node becomes feasible, because the
loop never stops retrying. Do not delete the pod to "re-trigger" anything.

## 1 · First command, always

```bash
kubectl describe pod <pod>          # read the Events section at the bottom
```

You are looking for a line of this shape, generated by the scheduler itself:

```text
Warning  FailedScheduling  2m  default-scheduler  0/6 nodes are available:
         2 Insufficient cpu, 3 node(s) didn't match pod topology spread
         constraints, 1 node(s) had untolerated taint {gpu=true}.
```

Read it as facts, not as an error: six nodes considered, and a count per rejection reason.
The counts must sum to the node total; if they do not, you are reading a truncated message.

- [ ] If there are **no events at all**, they were garbage-collected — the default TTL is
      one hour, so a pod Pending since yesterday shows nothing. Fall back to
      `kubectl get pod <pod> -o yaml` and read `.status.conditions`.
- [ ] `kubectl get events --sort-by=.lastTimestamp` when several things broke together —
      that is your timeline.
- [ ] `kubectl top nodes` needs metrics-server installed. On a bare cluster it errors, and
      that is not a symptom of anything.

## 2 · Decode the reason

| Reason string | What it means | Where to look |
|---|---|---|
| `Insufficient cpu` / `Insufficient memory` | The pod's **request** exceeds that node's *allocatable minus already-requested*. Actual usage is irrelevant. | §3, §4 |
| `node(s) had untolerated taint {k=v}` | The node is refusing you. Taints are the one mechanism where the node pushes back; a toleration is your explicit statement that you can handle it. | §5 |
| `node(s) didn't match Pod's node affinity/selector` | You asked for node properties nothing has. Check for a typo'd label or a retired instance type. | §5 |
| `node(s) didn't match pod anti-affinity rules` | A **hard** anti-affinity with fewer feasible nodes than replicas. Capacity is usually irrelevant here. | §3 |
| `node(s) didn't match pod topology spread constraints` | `whenUnsatisfiable: DoNotSchedule` and placing you would exceed `maxSkew`. | §3 |
| `node(s) were not ready` / `unschedulable` | Cordoned, draining, or the kubelet has stopped reporting. | §6, §7 |
| `node(s) had volume node affinity conflict` | The pod's PersistentVolume lives in a zone the candidate nodes are not in. Storage pinned you to a zone. | §5 |
| `Insufficient <extended resource>` | GPUs and similar. The device plugin may simply not be running on the node. | §5 |

## 3 · Is it a constraint or a shortage? (the fork that matters)

- [ ] **Free capacity present and still Pending → it is a constraint, not a shortage.**
      This lesson produced exactly that case: a replica Pending on a cluster holding
      **10,250m of free CPU while requesting 250m**, because a hard anti-affinity allowed
      one replica per node and there were only three nodes. Adding nodes fixes a shortage
      and does nothing for a constraint.
- [ ] Count feasible nodes against replica count. A hard anti-affinity caps you at one
      replica per node — a Deployment with more replicas than nodes will *always* leave the
      remainder Pending, by design.
- [ ] Check `whenUnsatisfiable`. `DoNotSchedule` is a hard guarantee that can leave replicas
      unschedulable; `ScheduleAnyway` turns your spread constraint into a suggestion, and the
      scheduler will pile replicas onto one node exactly when the cluster is tight — which is
      exactly when you needed the guarantee. Know which one you chose.
- [ ] Decide deliberately: relax the constraint, or add capacity in the domain the constraint
      is expressed over (more *nodes* for anti-affinity, more nodes *in the deficient zone*
      for a zone spread).

## 4 · The requests arithmetic

The scheduler subtracts **requests**, never usage. `kubectl top` showing 30% utilisation
tells you nothing about schedulability.

```bash
kubectl describe node <node>   # Allocated resources: requests and limits vs allocatable
```

- [ ] Compare the pod's request against **allocatable**, not capacity. The kubelet and
      system reservations come off the top.
- [ ] Work out which resource **binds**. In this lesson a 4000m / 8192Mi node fit 8 replicas
      by CPU and 16 by memory, so CPU was the binding constraint — and knowing which of your
      two resources binds is most of capacity planning.
- [ ] A request larger than any single node's allocatable is unschedulable forever, however
      many nodes you add. Check this before scaling the pool.
- [ ] **Every workload has requests set.** A pod with no request is scheduled as if it were
      free: the node accepts it and every other pod there now competes with a ghost. If you
      set exactly one thing today, set requests.
- [ ] Requests are what the *scheduler* reserves; limits are what the *kernel* enforces as
      cgroup v2 values on the node. They are different numbers with different enforcers:
  - [ ] over `cpu.max` → **throttled**. No error, no exception, no log line — just p99
        latency nobody can explain. Set CPU requests always, CPU limits rarely.
  - [ ] over `memory.max` → **killed**. Uncatchable OOM, **exit code 137**, nothing in your
        logs. Set memory requests and limits both; memory is incompressible.
  - [ ] A memory limit equal to your steady-state usage is a timer. One large response body
        crosses it.

## 5 · Node-side refusals

- [ ] `kubectl describe node <node>` → check `Taints`. GPU pools, dedicated hardware and
      draining machines all keep general traffic off themselves this way.
- [ ] Direction check: **affinity is the workload choosing a node; a taint is the node
      refusing workloads.** If you are editing the wrong side, nothing will change.
- [ ] Verify node labels actually exist, exactly as spelled, before blaming the scheduler:
      `kubectl get nodes --show-labels`.
- [ ] For a volume node-affinity conflict: the PersistentVolume is zone-pinned and the pod
      must follow it. Either the pod goes to that zone or the workload needs storage that is
      not zone-local.
- [ ] Add a toleration only when you mean it. A blanket toleration re-admits general traffic
      onto nodes that were deliberately reserved.

## 6 · A node died — what should have happened, and when

- [ ] Expect recovery to be automatic and to take **about five and a half minutes on a stock
      cluster**: roughly **40 seconds** of missed kubelet heartbeats before the node
      controller marks it `NotReady`, then a default **300-second** toleration for
      `node.kubernetes.io/unreachable` before pods are evicted and rescheduled. This is
      deliberate — it stops a network blip evacuating a healthy machine. If your business
      needs less, that is a number you change and test, not a bug.
- [ ] Do not open a runbook for a single node failure that is inside that window. The loop
      is converging; the next observation simply disagreed with declared state.
- [ ] If it is *outside* the window, the interesting question is whether the replacements are
      **Pending** — in which case you are back at §1, and the failure exposed that you had no
      spare capacity, not that recovery is broken.
- [ ] Measure the blast radius you actually took, and compare it against your placement
      policy. Same cluster, same failure: bin-packed cost **66.7% of serving capacity** with
      3 ticks to recover and 20 replica-ticks of lost work; node-spread cost **16.7%**, 2
      ticks and 4 replica-ticks — **4.0x the blast radius, 5.0x the lost work**, decided by a
      scoring preference set weeks earlier.
- [ ] If a whole **zone** went, node-spread is not the protection you thought: it lost
      **50%** here because zone-1 held 3 of the 6 nodes, against **33.3%** for zone-spread
      and **100% (with 2 ticks at literally zero capacity)** bin-packed. Spread over
      `topology.kubernetes.io/zone`, with `whenUnsatisfiable: DoNotSchedule`.
- [ ] **A failure domain is whatever fails together.** Write down what yours are — node,
      rack, zone, region, and also shared dependencies like one NAT gateway or one registry.

## 7 · A drain will not finish

- [ ] A PodDisruptionBudget constrains **voluntary** disruptions only — drains, upgrades,
      autoscaler scale-downs, anything calling the Eviction API. It cannot stop a power
      supply failing, and it was never meant to.
- [ ] The classic deadlock: `minAvailable: 3` on a workload with `replicas: 3` means no pod
      may ever be evicted, so drains hang forever and someone deletes the PDB mid-incident.
      Express budgets as **`maxUnavailable: 1`**, or a `minAvailable` strictly below the
      replica count.
- [ ] Re-check the arithmetic against your *minimum* replica count, not your current one. An
      autoscaler scaling you to 3 at 03:00 can turn a satisfiable budget into an
      unsatisfiable one.
- [ ] If a drain is stuck, `kubectl get pdb` and compare `ALLOWED DISRUPTIONS` against 0
      before touching anything else.
- [ ] A drain also stalls on pods with no controller (bare pods) and on unreplicated local
      storage. Those need `--force`, and forcing means accepting the loss.

## 8 · Things not to do

- [ ] **Do not delete a Pending pod to re-trigger scheduling.** It is already retried every
      pass. You will get an identical pod with a new name and lose the events.
- [ ] **Do not SSH to a node and inspect the container runtime to find out what should be
      running.** That tells you what *is*, not what *should be*, and the difference between
      them is the only interesting quantity in the system. The API server is the source of
      truth.
- [ ] **Do not raise the replica count to compensate** for replicas stuck Pending. The new
      ones will be Pending too, and you have made the anti-affinity or spread arithmetic
      worse.
- [ ] **Do not mutate a Deployment's `selector`.** It is immutable for good reason — changing
      it makes the loop disown its existing pods and create a full new set beside them.
- [ ] **Do not add a CPU limit to fix a latency problem.** Throttling is a latency problem.
- [ ] **Do not delete a PDB to unblock a drain** without a plan to restore it. That is the
      moment the budget existed for.

## 9 · Prevention — the pre-ship items

- [ ] **Requests set on every container.** Memory requests and limits both; CPU requests
      always, CPU limits only with a throttling metric on a dashboard.
- [ ] **Topology spread over the domain that actually fails together**, with
      `whenUnsatisfiable: DoNotSchedule`, and the replica count checked against the number of
      domains so the constraint is satisfiable.
- [ ] **A PDB on every workload with more than one replica**, written as `maxUnavailable`.
- [ ] **Headroom for one whole failure domain.** If losing a zone leaves you Pending, you did
      not have a spread problem, you had a capacity problem that spreading revealed.
- [ ] Probes configured per the health-checks contract: readiness fast and reversible,
      liveness slow, forgiving, and checking **nothing external**.
- [ ] `preStop` sleep plus a `terminationGracePeriodSeconds` that exceeds preStop + longest
      request + flush.
- [ ] Images referenced **by digest**, so a Pending-to-Running transition cannot also change
      what code runs.
- [ ] A dashboard for `Pending` pod count and pod age in Pending. A pod Pending for a day
      generates no alert on its own, and its events expire in an hour.

> ## Decision shortcuts
>
> **"Is there free capacity?"** Yes and still Pending → a constraint, not a shortage. Adding
> nodes will not help.
>
> **"Which number does the scheduler use?"** Requests. Always requests. Usage never enters
> the filter phase.
>
> **"Is this disruption voluntary?"** If a human or an autoscaler initiated it, a PDB
> applies. If hardware initiated it, no PDB in the world will help and you are relying on
> replica count and spread.
>
> **"Am I reading what is, or what should be?"** A node tells you what is; the API server
> tells you what should be. If they disagree, the loop is mid-convergence — or something
> outside the loop changed the world, which is drift wearing a different hat.
