# Capstone: A Suite That Catches Real Bugs

> Nine layers of testing — types, unit, fakes, integration, determinism, contract, async, property-based, fault injection — 52 checks, **195 seconds of CI**, and every technique this phase spent fourteen lessons building. Run against 31 real bugs seeded into one order service, it catches **28**. Three survive everything, and one of them is caught by a **fourteen-line check that uses no technique at all** — it just asserts an invoice somebody worked out from the spec by hand, and it kills **7 of the 31** on its own. Then the result that dissolves every "unit vs integration" argument you have ever had: static typing is worth **2 bugs when you build it first and 0 when you build it last**, the same nine layers reordered give **four different answers**, and the exact order-free ranking agrees with build order on **1 of 9 positions**.

## The Problem

It is 09:40 on a Monday and you have inherited a service.

It is not large. An HTTP handler that prices an order, a repository over a real database with four migrations, an async worker draining a queue, and an outbound call to a payment provider. Four hundred lines of production code. It has been in production for eight months, it has **no tests at all**, and the team that wrote it left.

You have this quarter to make it safe to change. You also have fourteen lessons of technique and a manager who asked, in the standing meeting, a question you could not answer:

> "Fine — but what do we build *first*, and how will we know when to stop?"

The honest answer is that nobody in this industry knows, because almost nobody measures it. The literature ranks test types by argument. The pyramid says unit tests, mostly, because they are cheap. The trophy says integration, mostly, because it is realistic. Both are stated as universal facts about testing and neither is accompanied by a number from your repository.

So do the thing the phase has been building toward. Seed the service with **31 bugs**, drawn from every class these fourteen lessons covered — a `>=` that became `>`, a migration that forgot its backfill, a field that changed meaning on the wire, a redelivered event, a lost update, a renewal date computed on the box's local clock, a retry that charges twice, a handle that leaks on the error path, a dependency that goes slow rather than down. Then build the suite one layer at a time and, after each layer, ask the only question that has an answer: **which bugs does this layer catch that no previous layer caught?**

That is an ablation, and the output is not a suite. It is a **31 × 9 matrix** and the marginal-value table you can read off it — the thing you can take into the standing meeting.

Three warnings about what you are going to find, because all three surprised me while measuring.

The marginal values **are not a property of the layers**. They are a property of the layers *and the order you built them in*, and reordering changes them by up to a factor of infinity — 2 bugs becomes 0. There is no context-free ranking of test types, and this lesson will show you the numbers that prove it.

Some bugs are caught by **nothing**. Not by a weak version of the nine layers — by the strongest version of all nine at once. And the reason is not effort.

And the single most valuable check in the whole file is not in any of the nine layers.

## The Concept

This section is short. Almost nothing here is new; it is the method for combining what you already have.

### Ablation: the only honest measure is marginal

The instinct is to score each layer on what it catches. Do that here and you get: types 2, unit 5, fakes 2, integration 7, determinism 6, contract 5, async 5, property 3, fault 7 — which **sums to 42 against a union of 28**. The layers overlap **1.50×**. Adding up "what each catches" double-counts more than a third of your suite and produces a ranking in which every layer looks worth having.

Ablation asks the counterfactual instead: build the layers in order, and after each one, count only the bugs that were *not already caught*. This is the same discipline [Lesson 01](../01-why-tests-exist/) applied to CI gates — where review and staging both scored well in isolation and contributed **0 marginal** each — and the same discipline [Phase 11's capstone](../../11-scalability-and-reliability/14-capstone-survive-the-region-loss/) applied to reliability defences by switching each one off. A number that does not survive the counterfactual is not a measurement, it is an argument.

### Every bug is a branch, not a row in a table

The matrix is only worth reading if the cells are real. Every one of the 31 bugs here is a live branch in live code — `subtotal > floor` instead of `subtotal >= floor`, a `CREATE TABLE` with the `UNIQUE` removed, a serializer that divides by 100 — switched on one at a time. Every layer is an ordinary check that raises when it observes something wrong, written the way a competent engineer writes that layer from the requirements. **No check knows the bug list.** That constraint is the entire experimental design; without it the matrix is a table of opinions with monospace formatting.

### A bug class is a hard capability boundary

[Lesson 02](../02-the-shape-of-a-test-suite/) measured that a unit test reaches **34.0%** of a service's defect population and that wiring defects are **zero at the unit level by construction** — a unit test's scope is one function and a wiring defect lives on the edge between two. The 11 classes used here exist for that reason: they are not severity buckets, they are *reachability* buckets. Whether a layer can catch a class is decided by scope and environment before anybody writes an assertion, and no assertion quality, coverage target or test count moves a hard zero.

### Reading a cost-per-bug table without fooling yourself

Cost per bug caught is `layer seconds ÷ marginal bugs`, and it has one trap: the denominator is order-dependent, so the ratio is too. Quote it only alongside the ordering that produced it. The useful companion is the **drop-one** number — how much detection you lose by removing exactly this layer while keeping the other eight — because that is the decision you actually make when a suite already exists. The two answer different questions and they disagree here by design.

### Marginal value is path-dependent

This is the point of the lesson. Marginal value is a set function evaluated along a path, and set functions with overlap are not additive. The first layer to reach a bug gets the whole credit; every later layer that also reaches it gets nothing. Build order therefore *assigns* value, it does not *discover* it.

There is exactly one order-free alternative and it has been sitting in game theory since 1953: the **Shapley value** (Shapley, "A Value for n-Person Games", *Contributions to the Theory of Games II*, Annals of Mathematics Studies 28, 1953) — a layer's average marginal contribution over *all* orderings. It is the unique attribution that is efficient (the values sum to the true total), symmetric, and gives zero to a layer that adds nothing anywhere. With nine layers there are 9! = 362,880 orderings, but the value only depends on subsets, so 2⁹ = 512 evaluations compute it exactly.

### What to do when a layer catches nothing

Two of the nine layers here have a drop-one value of **0**: remove them from the finished suite and detection does not move. The wrong conclusion is that they are worthless. The right one is that they are **currently redundant** with the layers around them — a statement about this suite today, not about the technique. It is also reversible, and cheaply: the redundancy is with the *most expensive* layers in the file, which is exactly why the budget optimiser buys those two first when CI seconds are scarce and drops them when they are not.

### The oracle problem, which no amount of testing reaches

A test compares what the code did against what it should have done. The second half is the **oracle**, and it has to come from somewhere outside the code. When it does not — when your expected value is another call to the thing under test — the assertion is a tautology: it cannot be false, so it cannot fail. [Lesson 10](../10-contract-testing/) measured the extreme case, a field redenominated from cents to dollars that round-tripped **400/400 in both directions** with a verdict of `FULL` while the consumer computed 1,857,277 where the provider meant 185,747,545. Contract testing is structural and stops at meaning. So does every other layer here, and the three survivors are all the same shape.

One `Svc` object carries the bug set, and `on()` is the only route a bug takes into code:

```python
def tier_bps(svc: Svc, subtotal: int) -> int:
    bps = 0
    for floor, b in TIERS:
        hit = subtotal > floor if svc.on("B01") else subtotal >= floor
        if hit:
            bps = b
    return bps
```

The suite never sees `Svc`. It reads a module-level `SUT` that the harness rebinds — exactly as a real suite does, by testing "the service" rather than parameterising itself over a fault list:

```python
def run_layer(fns, bugs: frozenset[str]) -> list[str]:
    """Run one layer against a service carrying `bugs`. Returns the failing names."""
    global SUT
    saved = SUT
    SUT = Svc(bugs)
    failures = []
    try:
        for fn in fns:
            try:
                fn()
            except Exception as exc:
                failures.append(f"{fn.__name__}[{type(exc).__name__}]")
    finally:
        SUT = saved
    return failures
```

Before any of it means anything, the suite has to be green on a healthy service. It is: **52 of 52 checks pass with zero bugs switched on, a 0.0% false-positive rate.** Every filled cell below is a bug being observed rather than noise, and that sentence is the load-bearing one — a matrix built on a flaky suite measures the suite's mood.

Read the columns. **No layer exceeds 7 of 31 on its own** — not integration, not fault injection, not the unit tests everyone argues about. The most any single technique reaches is 22.6% of the bug population, which is the quantitative version of the thing every experienced engineer says and nobody prices: you cannot test your way out with one kind of test.

Three rows are worth pausing on. **S02** — an `idempotency` table declared without its `UNIQUE` constraint — is caught by **five** layers, because a missing database constraint leaks into everything downstream of it: the contract suite over the repository port, the integration tests, the shuffled-order run, the duplicate-delivery test and the interleaved-claim test all trip on it. **T01**, a renewal date computed from the box's local clock, is caught by **one** layer and only one, because it is invisible unless something deliberately varies the hour: sweeping all 24 hours of a day gives two different renewal dates, and [Lesson 08](../08-determinism-time-randomness-order/) measured the general version of this at **1,454 wrong answers across the 8,784 hours of 2024**. And **B03**, a coupon minimum that became exclusive, is caught by **one** layer and it is not the unit tests — the hand-written unit test picks 25,000 against a 20,000 minimum because that is what a person types, and only the boundary-biased generator in the property layer lands on `subtotal == min_cents`.

Build the identical nine layers in reverse and static analysis is worth **0** instead of 2, the fakes-plus-contract-suite layer is worth **0** instead of 2, contract tests jump from 3 to **5**, fault injection from 5 to **7**, and property-based testing from 1 to **3**. Build them cheapest-first and determinism controls go from 2 to **4**. Every ordering catches the same **28** — the union is a property of the layers — but **eight of the nine layers change value depending only on when you built them.**

The mechanism is not subtle once you see it: the first layer to reach a bug takes the whole credit. Static typing catches `total_cents` serialized as a string and a total kept in a `TEXT` column. So does the contract layer. Whichever you build first is worth two bugs and whichever you build second is worth zero, and the *only* thing that determined which is your build order. This is the same finding [Lesson 01](../01-why-tests-exist/) hit from the other end, where reversing gate order moved staging from **0 to 18** and unit tests from **25 to 1**.

So compute the order-free answer. Coverage is a set function over 512 subsets, so the exact Shapley value of every layer is a few thousand arithmetic operations:

```python
def shapley(caught) -> dict[int, float]:
    """Exact Shapley value of each layer over the coverage set function."""
    cov = {frozenset(s): coverage(caught, s)
           for r in range(NL + 1) for s in itertools.combinations(layers, r)}
    out = {}
    for n in layers:
        rest = [m for m in layers if m != n]
        total = 0.0
        for r in range(NL):
            w = math.factorial(r) * math.factorial(NL - r - 1) / math.factorial(NL)
            for sub in itertools.combinations(rest, r):
                s = frozenset(sub)
                total += w * (cov[s | {n}] - cov[s])
        out[n] = total
    return out
```

Fault injection **5.53**, integration **4.28**, async **4.20**, unit **3.83**, contract **3.75**, determinism **3.12**, property **1.83**, types **0.75**, fakes **0.70**. They sum to exactly **28.00**, which is the efficiency property doing its job and a free correctness check on the implementation. And the ranking **agrees with the build-order ranking on 1 of 9 positions.**

Two readings of that, and you need both. The Shapley value is the right number for *crediting* a layer, because it is the only one that does not depend on an arbitrary path. The build-order marginal is the right number for *deciding what to build next*, because you are standing at a specific point on a specific path and the counterfactual you face is the one in front of you. Confusing them is how a team ends up importing another organisation's pyramid.

**C05** takes `total_cents` and divides it by 100 at the serialization boundary — right name, right type, wrong unit. Layer 1 checks the declared type and sees an `int`. Layer 6 verifies the consumer's contract, whose matcher is `integer`, and passes. Layer 4 posts an order and gets it back, and both directions are divided by the same 100, so they agree perfectly. The property layer's scope is the pure core, which never sees the wire. Measured: **the wire says `total_cents=206` where the invoice is 20662** — a 100× understatement in a correctly typed integer with the right name, passing **9 of 9** layers.

**M01** computes tax on the gross subtotal instead of the discounted net. The finance spec says net; the code reads the same paragraph the other way. This one is the most instructive, because the suite *does* exercise it — and here is the measurement, taken by walking every check's AST:

| Measured across all 52 checks, by walking their ASTs | Count |
|---|---|
| put a **non-zero discount** through the pricing core | 16 |
| assert an **exact value** of `tax_cents` or `total_cents` | 4 |
| do **both** — `l3_fake_and_real_agree_on_a_priced_order`, `l4_persisted_total_matches_the_computed_invoice`, `l8_property_components_reconstruct_the_total` | 3 |
| ...of those, compare it against a number **the test supplied** | **0** |

Sixteen checks price a discounted order. Four assert an exact money value. Three do both — and **all three compare one output of the code against another output of the same code**: persisted versus computed, real repository versus fake, components versus total. A tautology cannot be false, so it cannot fail. M01 does not live in a missing test. It lives in a suite with **no independent oracle**, and it bills **15750 where the spec says 15225** — 525 cents of tax on money the customer never paid, on every discounted order.

**N01** drops an event whose currency the service does not recognise, instead of dead-lettering it. Measured: **all 52 checks use exactly one currency literal, `USD`, and the service knows two.** No fixture in the suite ever produces a third, so the branch is never executed. This is not a bug a test process can reach, because only production supplies the input.

Now the part that changes what you do on Monday. Add one more check — not a tenth technique, just an ordinary end-to-end assertion whose expected values were worked out from the finance spec by somebody who did not write the code:

```python
def probe_golden_invoice() -> None:
    """20000 gross; 5% tier = 1000; coupon 5000; net 14000; 8.75% of 14000 = 1225."""
    _, repo, app, _, _ = make_service()
    repo.migrate()
    c = Coupon("SAVE50", 0, date(2026, 12, 31), 5_000)
    _, body = app.post_orders({"customer_id": 1, "subtotal_cents": 20_000,
                               "currency": "USD"}, c, NOW)
    got = (body["subtotal_cents"], body["discount_cents"],
           body["tax_cents"], body["total_cents"])
    assert got == (20_000, 6_000, 1_225, 15_225), got
```

**Fourteen lines, one HTTP call, no new infrastructure — and it catches 7 of the 31 bugs on its own: W01, W03, S03, C01, C04, C05 and M01, including two of the three that nine layers and 52 checks could not.** It is not clever. It contains no technique. Its entire power is that the four numbers on the right-hand side came from the specification instead of from the implementation, which makes it the only assertion in the file that can disagree with the code about what the code is *for*.

N01 is not on that list and never will be. Its remedy is a monitor — `events_received` minus `events_settled`, alerting on the gap — which [Phase 9](../../09-logging-monitoring-and-observability/09-slis-slos-and-error-budgets/) builds and which would have caught it in minutes. A human, a spec, and a monitor: three survivors, three different answers, none of them a test.

### Mutation scoring, and what it is actually measuring

The program mutates a transcription of the pricing core with an AST rewriter — comparison flips, arithmetic flips, integer constants +1 — and runs the exact cases layers 2 and 8 use against each mutant. **15 mutants, 13 killed, 2 survived, a mutation score of 86.7%.** [Lesson 13](../13-coverage-and-mutation-testing/) measured the same technique over a larger target and found the ceiling is not 100% either: **three survivors that were confirmed equivalent by hand, putting the maximum at 95.7%.**

And the number that places it: **mutation testing detected 0 of the 31 seeded bugs.** That is not a criticism, it is the definition. It does not test the product; it tests whether the tests can see. It belongs in a nightly job and never in the detection matrix.

### Flake-adjusted trust, which is what the suite is actually worth

A suite that catches 28 of 31 bugs and is red half the time on clean commits catches nothing, because nobody reads it. Applying the per-layer flake rates from [Lesson 09](../09-flaky-tests/)'s ranges to the measured check counts:

| Suite size | Checks | P(green on a clean tree) | P(real bug \| red) | Evidence |
|---|---|---|---|---|
| **as built** | 52 | **79.42%** | **18.77%** | **+1.908 bits** |
| 5× | 260 | 31.60% | 6.50% | +0.378 bits |
| 10× | 520 | 9.99% | 5.02% | +0.005 bits |
| 20× | 1,040 | 1.00% | 4.58% | **−0.126 bits** |

At its current size a red build carries **1.908 bits** — genuinely informative. Grow the same nine layers ten-fold, which is what happens over two years of ordinary feature work, and a red build carries **0.005 bits**: it has stopped distinguishing a broken tree from a clean one. At twenty-fold it goes **negative** — a red build is now weak evidence *against* a real bug, because clean trees are red so much more often than buggy ones are rare. **At 10× the check count, layer 9 alone is red 78.1% of the time on a clean tree**, and it is also the most expensive layer to run. That is the compounding [Lesson 09](../09-flaky-tests/) priced at `P(green) = (1−f)^n`, arriving on a suite you just built.

| Budget | Bugs | of 31 | CI s used | Layers the optimiser keeps |
|---|---|---|---|---|
| 15 s | 12 | 38.7% | 14.26 | **1 types**, 2 unit, **3 fakes**, 5 determ |

| **90 s** | **22** | **71.0%** | **67.85** | **2 unit, 4 integr, 5 determ, 6 contract, 7 async** |

**Under 90 CI seconds, keep unit, integration, determinism, contract and async: 22 of 31 bugs for 67.85 seconds — 79% of the detection for 35% of the time.** The gate does not spend its whole budget, because the next layer it could afford does not fit.

Now read the 15-second row against the 90-second row, because that is the path-dependence result showing up as money. At 15 seconds the optimum **buys** layers 1 and 3. At 90 seconds it **drops** them. Nothing about typing or fakes got worse; layers 4 and 6 became affordable and cover the same four bugs more thoroughly. The optimal suite is not a ranking of techniques you apply everywhere — it is a function of your budget, and it is not monotone in it.

The frontier also flattens: **28 bugs at 185.05 seconds, while the whole nine-layer suite costs 195.36.** The last **195.36 − 185.05 = 10.31 seconds** buy exactly zero additional bugs, because layers 1 and 3 are completely redundant once 4 and 6 are present. Which brings the drop-one table, and the honest way to say that:

| # | Layer | Bugs lost if dropped | CI s freed | bugs/s |
|---|---|---|---|---|
| 1 | types / static analysis | **0** | 8.75 | 0.0000 |
| 2 | unit tests over the core | 3 | 0.70 | **4.3103** |
| 3 | fakes + a contract suite | **0** | 1.56 | 0.0000 |
| 4 | integration, real database | 3 | 34.80 | 0.0862 |
| 5 | determinism controls | 2 | 3.25 | 0.6154 |
| 6 | contract tests at the seam | 3 | 12.60 | 0.2381 |
| 7 | async, duplicates, reorder | 4 | 16.50 | 0.2424 |
| 8 | property-based on the core | 1 | 28.20 | 0.0355 |
| 9 | fault injection | 5 | 89.00 | 0.0562 |

A layer whose drop-one value is 0 is not useless — it is redundant *with this particular set of neighbours*, which is a different and reversible claim. Delete your static typing and your fakes on the strength of that row and the first day integration is too slow to run on every commit, you have lost the cheap copy of those four bugs. Keep them: at **8.75 and 1.56 seconds** they are 5% of the suite's runtime and they are the layers that still work when the expensive ones are switched off.

And the six bugs the 90-second gate lets through are not a failure of the gate. **B03, R01, R02, Y02, G01, G02** — a boundary only a generator finds, two races, and three overload behaviours — are the nightly job. A gate is not the whole suite; it is the part that has to answer before a human can reasonably be expected to wait.

## Use It

The nine layers map onto real tools, and the point of assembling them here is that every gate threshold below is justified by a number measured somewhere in this phase rather than chosen because it is round.

### `pyproject.toml`

```toml
[project]
name = "orders"
requires-python = ">=3.11"

[project.optional-dependencies]
test = [
  "pytest>=8.2", "pytest-asyncio>=0.23", "pytest-randomly>=3.15",
  "pytest-xdist>=3.6", "pytest-timeout>=2.3", "hypothesis>=6.100",
  "coverage[toml]>=7.5", "diff-cover>=9.0", "testcontainers[postgres]>=4.5",
  "time-machine>=2.14", "factory-boy>=3.3", "mypy>=1.10",
]

[tool.mypy]
strict = true
disallow_untyped_defs = true          # layer 1: the 2 bugs it catches cost 8.75 s

[tool.coverage.run]
branch = true                          # line coverage called an if-with-no-else 100%
[tool.coverage.report]
fail_under = 0                         # deliberately: gate the DIFF, not the total

[tool.hypothesis]
derandomize = false                    # a new seed finding a new bug is not a flake
```

`fail_under = 0` on the whole repo is deliberate and it is the single most contested line in the file. [Lesson 13](../13-coverage-and-mutation-testing/) measured a six-test suite with **not one assertion** scoring **100.0% line coverage** and killing **0 of 70** faults. A total-coverage gate is a target, and a target buys assertion-free tests — measured, **6 of 8** slots in the coverage-maximising suite went to tests that assert nothing. Gate changed lines instead, where an uncovered line is real information.

### `pytest.ini`

```ini
[pytest]
addopts = -q --strict-markers --timeout=60 -p no:randomly
asyncio_mode = auto
markers =
    unit: layer 2 - pure core, no I/O
    fakes: layer 3 - the shared contract suite, run against fake AND real
    integration: layer 4 - real database, session-scoped container
    contract: layer 6 - consumer contracts verified against this provider
    async_delivery: layer 7 - duplicates, reordering, DLQ
    property: layer 8 - hypothesis
    fault: layer 9 - toxiproxy, nightly
    quarantine: expires YYYY-MM-DD - MUST carry a date and an owner
```

`-p no:randomly` in the default `addopts` and `-p randomly` in the dedicated shuffle job, rather than shuffling everything always: [Lesson 08](../08-determinism-time-randomness-order/) measured that a 200-test suite gives up *some* order dependency in **3 shuffled runs** at 99% confidence but its rarest needs **182**, so shuffling belongs in a job that runs often, not in the one gating a merge.

### The CI workflow, and the job graph

```yaml
name: ci
on: [pull_request]
jobs:
  gate:                      # the 90-second budget, solved above: layers 2,4,5,6,7
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }             # diff-cover needs the merge base
      - run: mypy src/                       # layer 1 - 8.75 s, cheap copy of 2 bugs
      - run: pytest -m "unit or fakes" -n auto
      - run: pytest -m "integration or contract or async_delivery" -n 4
        # per-xdist-worker database: 4 workers on one DB were refused 4 of 4 times
      - run: coverage xml && diff-cover coverage.xml --fail-under=90
      - uses: actions/cache@v4
        with:
          path: .hypothesis/examples          # the regression database, NOT gitignored
          key: hyp-${{ github.sha }}
          restore-keys: hyp-                  # without a prefix this restores nothing
  shuffle:                   # order-independence, non-blocking, high frequency
    runs-on: ubuntu-latest
    steps:
      - run: pytest -m "unit or integration" -p randomly --randomly-seed=last
  nightly:                   # layers 8 and 9 - the 6 bugs the gate lets through
    if: github.event_name == 'schedule'
    runs-on: ubuntu-latest
    steps:
      - run: pytest -m "property" --hypothesis-profile=ci
      - run: pytest -m "fault"                # toxiproxy latency + abort toxics
      - run: mutmut run --paths-to-mutate src/pricing --use-coverage
```

The gates, and the number behind each one:

| Gate | Threshold | The measurement that justifies it |
|---|---|---|
| `mypy --strict` | zero errors | 8.75 s for 2 bugs; **0** if you already run layers 4 and 6, so keep it for the day you cannot |
| changed-line coverage | `--fail-under=90` | an uncovered line is `P(detect) = 0` exactly; a covered one measured **84.6%** (L13) |
| total coverage | **no gate** | a no-assert suite hit **100.0% lines and killed 0 of 70** (L13) |
| retries | `--only-rerun` on infra errors only | blanket `--reruns 2` reports a 3% race at **0.0027%** and hid **2 of 6** real races (L09) |
| suite green rate | ≥95% on clean commits | at **10× this suite's size** a red build carries **0.005 bits** |
| mutation score | ≥80% on money modules, nightly | 86.7% here; the ceiling is not 100% — equivalent mutants (L13) |
| golden invoice | **1 check, blocking** | catches **7 of 31**, including 2 of the 3 that nine layers missed |

### What to actually pick

Start with layers 2 and 5 — unit tests over a functional core plus a controllable clock and a fixed seed. Together they are **0.70 + 3.25 = 3.95 seconds** and **10 of the 31 bugs**, and layer 5 is what stops layer 2 from lying to you about dates. Add layer 4 next despite the price, because the database holds rules a double cannot: [Lesson 06](../06-integration-testing-real-database/) measured a hand-written in-memory fake enforcing **2 of 10** declared schema rules against the real engine's **10 of 10**. Then layer 7 if you have a queue, because duplicate delivery is not a rare event and [Lesson 11](../11-testing-async-and-event-driven/) measured a naive consumer over-crediting **$3,571.20** across 400 orders at an 8% redelivery rate. Layers 8 and 9 go nightly.

And whatever else you do, write the golden invoice. One check, an oracle from outside the code, blocking. It is the cheapest line in the entire table and it is the only thing in this file that can tell you the code is doing the wrong thing correctly.

## Key takeaways

- **Rank layers by what they add, never by what they catch.** The nine layers score 2, 5, 2, 7, 6, 5, 5, 3, 7 bugs individually — **summing to 42 against a union of 28, a 1.50× overlap.** And **no single layer exceeds 7 of 31 (22.6%)** on its own, which is the quantitative form of "one kind of test is not a strategy".
- **Marginal value is path-dependent, so no ranking of test types is portable.** The same nine layers and the same 31 bugs, in four build orders: static analysis is worth **2 bugs first and 0 last**, fakes the same, contract tests **3 to 5**, fault injection **5 to 7**, property-based **1 to 3**. Every ordering totals 28. Eight of nine layers move.
- **The Shapley value is the only order-free attribution, and it disagrees with build order almost everywhere.** Over all 512 subsets: fault **5.53**, integration **4.28**, async **4.20**, unit **3.83**, contract **3.75**, determinism **3.12**, property **1.83**, types **0.75**, fakes **0.70** — summing to exactly **28.00**, and **agreeing with the build-order ranking on 1 of 9 positions.**
- **The cheapest layer per bug is 203× cheaper than the dearest, and that is not an argument for deleting the dearest.** Unit tests cost **0.14 s per marginal bug**, fault injection **17.80**, property-based **28.20** — while fault injection has the *highest* Shapley value, because five bugs (two races, an unbudgeted retry, a missing timeout, a missing breaker) are reachable by nothing else at any price.
- **Three bugs survive all nine layers and all 52 checks, and none of them is a gap in effort.** A total redenominated into dollars passes **9 of 9** layers (`total_cents=206` where the invoice is 20662); tax on the gross survives because the **3** checks that both discount an order and assert an exact money value all compare the code against itself — **0** use a number the test supplied; and an unknown currency is dropped because all **52 checks use 1 currency literal** while the service knows 2.
- **The most valuable check in the file uses no technique at all.** Fourteen lines asserting one invoice hand-computed from the finance spec catch **7 of the 31 bugs**, including **2 of the 3** that nine layers missed. Its power is entirely that the expected numbers came from the specification instead of the implementation — an independent oracle, which is the one thing no amount of technique substitutes for.
- **A gate is not a suite, and the optimum is not monotone in the budget.** Solved exhaustively over 512 subsets: **90 CI seconds buys layers 2, 4, 5, 6, 7 — 22 of 31 bugs for 67.85 s, 79% of the detection for 35% of the time.** At 15 seconds the optimum *buys* layers 1 and 3; at 90 it *drops* them, because 4 and 6 became affordable and cover the same bugs.
- **Redundant is not useless, and it is a claim about today.** Layers 1 and 3 have a drop-one value of **0** and the frontier is flat from **185.05 s to the full 195.36 s** — the last 10.31 seconds buy nothing. At 8.75 and 1.56 seconds they are 5% of the runtime, and they are what still works on the day the expensive layers are too slow to run per-commit.
- **Suite growth is a trust decay you have already budgeted for by accident.** This suite is green **79.42%** of the time on a clean tree and a red build carries **1.908 bits**. At **10×** the check count that is **0.005 bits** and at **20×** it is **−0.126** — the same 90.3% detection, delivered through a channel nobody reads, with layer 9 alone red **78.1%** of the time.
